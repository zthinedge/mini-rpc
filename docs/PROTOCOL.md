# mini-rpc 协议层设计
## 协议层链路
TCP字节流->Buffer->RpcCodec解码->RpcMessage->Rpc业务层
## 1. 协议设计
在协议层设计之前，需要分析协议层需要什么。
首先根据流程进行分析，协议层最原始的需求是客户端把调用哪些方法，传入什么参数告诉服务端，服务端把执行是否成果，返回的结果告诉客户端
从业务使用场景进行分析，服务端需要知道客户端调用的是哪一个方法，所以客户端的请求必须携带“方位定位信息”，service_name、method_name，即路由信息；客户但需要传递业务参数，但是对于不同的方法，需要不同的参数，协议层不能根据每一个方法设计不同的字段，所以需要一段通用业务数据payload，表示具体对象序列化后的二进制数据；服务端返回结果，客户端需要知道调用是否成功，以及返回的结果是什么，所以response需要status_code error_text payload;请求和响应需要匹配，因为考虑到性能，所以可以一次发送多条请求，但是为了响应顺序很可能是乱序的，所以对于每一次调用需要一个唯一标识，即request_id;客户端调用可能会超时，但是这个只需要包括在本地就行了，不必写在协议中。再对TCP带来的技术约束分析，TCP是没有消息边界的，所以必须要接受端知道一条完整消息一共有多少个字节，meta_len payload_len ；可能连接错端口或者数据错位，比如RPC服务端可能会接收到http请求或者是错误协议，所以需要有协议标识magic用来判断是否符合mini-rpc格式；客户端和服务端可能会使用不同的协议格式，当客户端更新Header格式的时候，服务端还是旧的格式，可能会导致错误解析，所以需要加入版本号，也利于之后扩展；网络输入不可信，服务端不能无限等待或者无限分配内存，所以需要有本地限制，配置在解码器当中；多字节整数在不同机器上表示可能不同，所以协议需要规定统一使用网络字节序。根据以上的分析，request和response可以合并成rpcmessage，最后推导header，需要magic，version，messagetype，request_id，meta_len,payload_len,如果需要不同序列化格式可以加入codetype，需要压缩或者单向消息加入flags，属于扩展设计
